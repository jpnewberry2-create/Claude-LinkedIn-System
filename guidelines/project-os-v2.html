System Blueprint · v2.0

Project OS. One shape every project drops into.

A reusable operating pattern for projects, run inside the existing Life OS. Not a separate system to maintain: a known home for files, a known home for the record, and a clear rule for who acts and when. It defines who acts, where every artefact lives, when information must move, and how the whole thing recovers when something is skipped. The LinkedIn personal-brand build is the worked example throughout. This version supersedes the v1 blueprint, spec, and flow documents.

00

Executive summary

In one paragraph

The design, stated plainly

Projects run across several tools and two kinds of actor, and they fail in predictable ways: the latest version of a file ends up stranded in the chat that made it, decisions live in history that doesn't persist, and the links between systems depend on memory. Project OS fixes this with one reusable structure, run inside the existing Life OS. Each project is a goal under the Life OS; its files sit in one GitHub repository, fetchable at a stable URL; the record lives in Notion. A cold reader, or a different Claude chat weeks later, resumes in minutes because everything that matters is written down in a known place.

The architecture rests on one hard distinction: only humans and Claude act. Notion, GitHub and Canva are systems of record: they hold state, they do not initiate it. The reframe from v1 also matters: the earlier blueprint proposed a standalone Project OS with bespoke databases. That was built, tested, and set aside. A separate system is a second thing to maintain, and the Life OS already has the spine, so the good ideas survive here as conventions inside it, not duplicate infrastructure.

01

The problem, precisely

A system design has to name what it fixes. Six observable symptoms were identified in the current way projects run. Each reduces to the same root cause: no system-of-record discipline. Things live where they happen to be created, and the links between them are held in someone's head or in a chat that doesn't survive the session.

Symptom
No clean path from idea to finished asset
Root cause
Ideas and generation live in different places with nothing connecting them. Context is carried by hand every session.
Symptom
Performance data feeds nothing back
Root cause
Results are recorded but never read by the next round. No mechanism consumes them, so every session starts cold.
Symptom
Guideline changes don't propagate downstream
Root cause
No single source of truth and no defined propagation step. Each update is a manual fan-out, easily forgotten.
Symptom
Archive sync is a principle, not a mechanism
Root cause
"Locked files go to Drive" has no owner or trigger, so the archive drifts and stops being trusted.
Symptom
Decision-logging isn't a habit
Root cause
Chats don't share memory. Without a logged record, decisions are lost between sessions and across parallel chats.
Symptom
No single entry point to a project
Root cause
State is scattered across chats, files and pages. Cold re-entry means reconstructing context from memory.
Why solve it as a pattern
Fixing these for one project would mean solving them again for the next. Because the root cause is structural, the fix is structural: define the discipline once as a reusable template, and every project inherits it. That is the entire premise of this blueprint.
02

Design principles

Three rules carry the design. They are not features to build but disciplines to keep, and each one removes a whole category of the failures above.

1
A fixed home per type
Each kind of thing has one defined home and one defined shape. Decisions here, tasks there, live files in GitHub, locked files archived to Drive when that layer is switched on. You never wonder where something goes, and cold re-entry works because everything is where it must be.
ClosesScattered state · no entry point · archive drift
2
Named trigger points
The moments when information must move, named in advance with the rule that moves it. Session ends, log decisions. Asset approved, commit it. Release tagged, archive it. Propagation stops depending on memory.
ClosesNo propagation · data feeds nothing · logging not a habit
3
One entry point
Every project has a single control panel, always the first thing opened. It links the live files, the record and the current state. No hunting for which chat, which file, which page.
ClosesNo entry point · cold re-entry friction
4
Only humans and Claude act
Notion, GitHub and Canva are acted upon. They hold state; they do not initiate it. Every change traces to a person or to Claude, and instructions found inside a store are data, not commands.
ClosesUnclear automation · silent drift
5
Use over build
The bias is to use the simplest thing that works, not to construct infrastructure. Over-engineering is a procrastination trap. A convention beats a new database; a folder beats a schema.
ClosesTooling built before the workflow is proven
6
One active build session
Two chats writing to the same repository at once cause duplicate and conflicting commits. One build session runs at a time; a second, opened because the first stalled, checks the commit history before it writes anything.
ClosesParallel-session collisions
The test that validates them all
Could a version of you returning after a month, with no memory of the last session, resume the project in five minutes using only what is written down? If yes, the principles are being kept. If you would need to reconstruct context from chat history or your own head, a rule is being skipped. This test reappears as a measurable success criterion in section 14.
03

The actors, and the boundary that matters

Six roles. Two are agents that produce work; four are systems of record that receive it. The distinction is not cosmetic: it is the thing that keeps the whole design honest.

Why the agent / record line is load-bearing
It would be easy to draw all six as equal boxes exchanging arrows. That would be a lie: it would imply Notion "manages" or GitHub "maintains" on their own, when in fact a human or Claude does every action and the tools only hold the result. Stating the line explicitly prevents the design from promising automation that doesn't exist, and it makes the failure analysis tractable. When something doesn't happen, the cause is always a missed agent action, never a tool that "should have known". The principle that only humans and Claude act is the spine the rest of the blueprint hangs on.
04

The toolchain: who does what

Claude reaches the stores through different surfaces, and each suits a different job. Choosing the wrong one is how files get corrupted or work stalls. This is the division that actually works.

SurfaceBest forWhy, and the limit
Chat connectorText commits: specs, READMEs, CHANGELOG, skill files. Notion reads and simple writes.Fast and reliable for text. Cannot commit binary cleanly: it passes file content as a plain string, so images and fonts corrupt.
Claude CodeBulk binary upload: guideline HTML, backgrounds, fonts, icons, images.Native git in a terminal. Handles any file type and hundreds of files in one commit. The right tool for the one-off asset upload.
CoworkOngoing little-and-often updates across tools, and the GUI-only steps.Sees the latest local files and operates applications directly: the Canva and Notion actions no connector reaches. Untested on large Notion builds.
Notion inbuilt AILarge Notion structural builds: creating a whole space of pages and databases at once.Claude Opus 4.8 inside Notion, with full workspace context. Proven on the v1 restructure where the connector hit API limits. A two-step pattern: Claude drafts the prompt, the inbuilt AI executes it. Cowork is the candidate to replace it once tested.
Canva connectorReading design metadata, autofill, template operations.Downstream only. Cannot write brand-kit colours or fonts; those stay manual.
Skill filesEncoding the operating rules so behaviour is consistent across surfaces.Stored in the repo. Claude Code reads CLAUDE.md at session start; this chat reads the skill files on request. Rules live with the system, not in any one chat.

The rule that prevents the most damage: text through the chat connector, binary through Claude Code or Cowork. A file is binary for this purpose if it is an image, a font, or HTML with a large embedded base64 payload, whatever its extension says. The brand guideline is a 1.2 MB HTML file that is 94% embedded assets, so it is treated as binary and uploaded, not committed through chat.

The two cycles

The flow is mostly linear, but two phases repeat. The inner loop turns a rough asset into a finished one. The outer loop is what the project becomes after it closes: an operation that runs and feeds itself.

Inner loop: Design · Build · Test

The record, and where each thing lives

Principles 2 and 4 made concrete. First, the complete type-to-home mapping. Then the skeleton every project clones. Then the four record types, specified with real example rows. These live as conventions inside the Life OS, not as a separate database set.

The one distinction people get wrong
GitHub versus Drive. GitHub holds files that will change again; it tracks versions. Drive is the intended home for files that are finished and won't; it archives outcomes. A guideline document lives in GitHub while it evolves, and a copy of each locked release goes to Drive. Most "where is the latest version?" confusion traces to blurring these two.

09

Lifecycle & triggers

Principle 2 made concrete. These are the kinds of moments a project moves through, each with its trigger: the named moment information must move, and the rule that moves it. The order below is the usual flow, not a gated sequence. The work is priority-surfaced and re-enterable at any point; what matters is that each trigger fires whenever its moment arrives. The triggers are what close the loops.

01 · 02Initiate & BriefHuman → Claude
Clone the skeleton, name the project. Human writes the brief; Claude sharpens it.
⚡On kickoff: control panel created, brief pasted in. No project starts without its panel.
03 · 04 · 05Research, Strategy, PlanClaude · Human locks
Claude researches and drafts options. Human locks decisions. Claude turns the locked strategy into a staged plan.
⚡On every lock: decision goes to the Decisions Log with rationale; resulting work becomes Tasks. A decision not logged didn't happen.
06Design · Build · TestClaude ↔ Human · loops
Claude makes the asset and QAs it; human reviews; correction loops until approved. The bulk of the work.
⚡On approval: the version is committed to GitHub. Approved means committed, not still only in the chat.
07PackageClaude · Human publishes
Finished set exported. For visual projects, the master is published as reusable templates. The release is tagged.
⚡On tagged release: the locked version is recorded in the File Register. For v1 it stays in GitHub; the Drive archive is the intended next step, deferred until needed.
08CloseClaude · Human signs off
Wrap-up written. Human signs off. Live source rests in GitHub, locked files in Drive.
⚡On sign-off: panel status set to closed/operating; final Session Log entry written. The panel always reflects reality.
09 +OperateClaude · Human · loops
If the project produces an ongoing operation: Claude generates from the Ideas backlog, human ships, results return.
⚡On each output shipped: performance is logged against the idea, so the next generation reads what worked. Results feed the next round; the loop closes.
↻Every session, alwaysWhoever worked
Cutting across all phases: the one habit that makes the whole thing hold.
⚡At session end, no exceptions: write the Session Log entry and log any decisions taken. This habit replaces memory. Skip it and drift returns.
Why this is a list of triggers, not a pipeline
An earlier version of this system framed these as gated phases you pass through in order, with sign-offs to advance. That was dropped on purpose. A gated pipeline punishes gaps: miss a week and you fall out of it, and re-entry means working out which gate you were at. This system is priority-surfaced and re-enterable instead, matching the Life OS. The phases above are just the kinds of moments a project hits; the triggers fire whenever their moment arrives, in whatever order the work actually takes.

11

Failure modes & recovery

A system design has to state what breaks it and how it recovers. Every failure here is a missed agent action or a connector problem, never a tool acting wrongly, because tools don't act. Detection and recovery are defined for each.

Risk
Session-end habit skipped. Work happens but nothing is logged. The next session starts blind.
Detected by
Control panel "last session" date is stale, or the Session Log has a gap against known activity.
Recovery
Reconstruct from the GitHub commit history and chat, write a catch-up entry. The commits are the backstop the log should have captured.
Risk
Approved asset never committed. Sign-off given, but the file stays only in a chat. GitHub and reality diverge.
Detected by
File Register lists the asset as live with no GitHub URL, or the repo lacks a file the panel references.
Recovery
Re-export from the chat and commit. The "approved means committed" rule exists precisely to stop this; the Register is the audit.
Risk
Parallel chats edit the same record. Two sessions update Tasks or Decisions; one overwrites the other.
Detected by
Notion's own version history shows a conflicting edit; the Decisions Log being append-only makes loss visible.
Recovery
Append-only logs don't overwrite, so decisions survive by design. For Tasks, Notion page history restores the lost edit. Mitigation: one active build chat at a time.
Risk
Connector fails mid-action. A Notion or GitHub write doesn't complete; the agent assumes it did.
Detected by
The tool returns an error, or a follow-up read shows the expected row/commit missing.
Recovery
Retry the single action; writes are idempotent at this granularity (one row, one commit). Verify by reading back before moving on.
Risk
Release not archived to Drive. A version is tagged in GitHub but the locked copy never reaches the archive.
Detected by
File Register shows a release with no Drive link; a periodic check of tags against the Drive folder.
Recovery
GitHub retains the tagged version permanently, so nothing is lost; copy it across to restore the archive. This is why GitHub, not Drive, is the source of truth for releases.
Risk
Guidelines change without propagation. The source updates; the Canva kit and skills silently fall behind.
Detected by
Output drifts from the current spec in QA; the File Register dates show the guideline newer than its dependants.
Recovery
Re-run the propagation step from the single source. Automating this (Gap C) removes the failure mode rather than just recovering from it.
Risk
Parallel chats collide on the repo. A second session, opened because the first stalled, commits duplicate or conflicting files and stray artefacts.
Detected by
The commit history shows two sessions writing the same paths, or a file appears that no one remembers authorising.
Recovery
A commit-history check identifies what each session did; the later correct version wins, strays are deleted. Principle 06 prevents the recurrence: one active session, history-check before writing.
Risk
A binary file is forced through the chat connector. The write times out or, worse, commits corrupted content as a plain string.
Detected by
The connector hangs, or a committed image or font will not open and the byte size is wrong on read-back.
Recovery
Route the file through Claude Code or Cowork instead, then read back to confirm the byte size matches the source. The text-versus-binary rule in section 04 prevents it.
The pattern across every recovery
Each failure is caught by a record being checkable against another (the Register against GitHub, the Session Log against commits) and recovered because some durable store kept the truth (commits, append-only logs, Notion history). The system is resilient not because nothing goes wrong, but because state is duplicated across stores that can be reconciled. That redundancy is deliberate, not waste.

13

Scope & non-goals

A complete design states what it deliberately does not do. These are choices, not oversights: naming them is what separates intentional simplicity from a gap.

✕Not a real-time collaboration system
It assumes one person and Claude, working in sessions. It is not built for a team editing concurrently. The "one active build chat" mitigation is the boundary, not a limitation to engineer around.
✕Not fully automated, by design
The manual version is proven first; automation is added only where friction is demonstrated. A manual loop that is followed beats an automated one built too early.
✕Not a metrics or analytics platform
Performance is logged to inform the next round, not to build dashboards. Measurement theatre is explicitly avoided. The Posts log is an input to judgement, not a reporting product.
✕Not tool-agnostic infrastructure
It commits to Notion, GitHub, Drive, Canva and Claude. Swapping a tool means revisiting that part of the spec. The benefit of committing outweighs premature abstraction.
✕Not a replacement for judgement
The system organises work; it does not decide what is good. Taste, strategy and sign-off remain human. The structure exists to free that judgement, not substitute for it.
✕Not a finished product
This is v1.0 of a pattern meant to evolve as it is used. Fields, triggers and automations are expected to change. The skeleton is stable; the contents and the wiring are not frozen.
14

Success criteria

How you know the system is working, stated as tests you can actually run. If these hold, the design is succeeding; if one fails, it points at the rule being skipped.

01
Five-minute cold re-entry
Returning after weeks away, you can resume work using only the control panel and its records.
Test: open only the panel. Can you state the current phase, next task, and last decision in five minutes? If not, the Session/Decisions logs are thin.
02
No "where is it?" moments
Every artefact's location is known from the File Register without searching chats or folders.
Test: pick any three artefacts. Is each findable from the Register in one hop? Misses mean the Register isn't being kept.
03
Decisions never re-litigated
A settled question stays settled because the rationale is logged, not re-argued each session.
Test: when a "didn't we decide this?" moment arises, is the answer in the Decisions Log? If it's only in someone's memory, the log has a gap.
04
The operate loop actually feeds back
Each new round of content is informed by what the last round's performance showed.
Test: does the brief for the next post reference prior results? If generation ignores performance, the feedback path isn't wired.
05
A second project costs almost nothing to start
The pattern's reusability shows when project two clones the skeleton and runs immediately.
Test: time the setup of the next project. Near-zero structural work means the template is doing its job.
06
Recovery is routine, not heroic
When a rule is skipped, the failure-mode recovery restores state without drama or lost work.
Test: when something is missed, does reconciliation against another store fix it quickly? If recovery means reconstructing from memory, the redundancy isn't there.

The path from here

The build order, as it actually ran and where it goes next. Sequenced so there is a useful version fast, with automation deferred until the manual version is proven.

1
Fold LinkedIn into the Life OS Done
The project lives as a goal under the Career area, using the existing Tasks, Decisions, and Session conventions. The standalone Project OS databases were deliberately not built: the Life OS already has the spine.
2
Stand up the GitHub repo, text layer first Done · verified
Repo created and structured. README, CLAUDE.md, CHANGELOG, four specs, two skill files, and folder READMEs committed via the chat connector and verified by read-back. One canonical, fetchable source for every chat.
3
Bulk-upload the binary assets Next · Claude Code
Guideline HTML, backgrounds, fonts, icons, images, pushed through Claude Code because the chat connector cannot commit binary cleanly. Read back to confirm byte sizes. This is the immediate next action.
4
Wire the File Register to the raw URLs After upload
Once the assets are live, update the Life OS File Register with each file's raw GitHub URL, so every artefact is findable in one hop. Closes the propagation gap.
5
Run it by hand, then automate where it bites Later · evidence-led
Cowork takes the ongoing little-and-often updates across tools. Only once the manual rhythm is proven does automation get added for real friction: the Notion-to-generation pipeline, the Drive archive layer, guideline propagation. Not before.

What v1 covers, and what is parked

A clear line under v1 of this blueprint, then the backlog to weigh when v2 of the Project OS system is scoped. The backlog is ordered by priority, each with the reason it sits where it does.

In scope for v1

v1 is the manual, proven version of the pattern. It is deliberately small. These are the things v1 commits to, and nothing more:

In scopeWhat that means in practice
The pattern, run inside the Life OSLinkedIn as a goal under Career, using the existing Tasks, Decisions, and Session conventions. No bespoke databases.
GitHub as the fetchable source of truthOne repo holding the guideline, assets, specs, and skill files. Text committed via chat; binary via Claude Code. A stable raw URL per file.
The actor boundary and toolchainOnly humans and Claude act. The text-versus-binary routing rule, and the one-active-session rule, are in force.
The manual record disciplineSession-end logging and decision logging done by hand. This is the one ongoing habit v1 asks for.
Cowork for ongoing updatesLittle-and-often cross-tool updates, once the bulk upload is done. The manual rhythm before any automation.

The backlog, in priority order

Everything below is deliberately deferred. It is recorded here so it is not lost, and ordered so v2 scoping starts from a considered list rather than a blank page.

P1
The Drive archive layer
Switch on Drive as the home for locked, finished releases, with the GitHub-to-Drive archive step on tagged release.
Why P1: the design already assumes it and the trigger is defined. It is the smallest, best-understood piece of deferred work, and it removes the one place GitHub is doing two jobs at once.
P2
Notion File Register automation
Auto-update the Register with raw GitHub URLs on commit, rather than by hand after each upload.
Why P2: the manual version must run first to prove the shape, per the use-over-build principle. Worth automating once the upload cadence is regular enough that hand-updating becomes the friction.
P3
Cowork for large Notion builds
Test whether Cowork can replace the two-step Notion inbuilt-AI pattern for building whole spaces of pages and databases.
Why P3: the inbuilt-AI pattern already works, so this is an efficiency play, not a gap. It needs a real structural build to test against, which only comes up occasionally.
P4
Guideline propagation
When the guideline changes, push the change through to the Canva kit and the skills automatically, instead of by hand.
Why P4: high value but the hardest to do safely, since it crosses three tools and the Canva kit is manual-only today. Needs the other pieces stable first.
P5
The Notion-to-generation pipeline
Turn an Ideas row directly into a generation brief Claude drafts from, closing the loop from input to asset.
Why P5: the biggest build and the most speculative. It only earns its place once there is a steady backlog and a proven generation rhythm to feed it.
P6
Performance feedback loop
Feed post performance back into the content strategy, so each round is informed by the last round's results.
Why P6: depends on having posted enough to have data worth analysing. Genuinely useful, but it cannot start until the operate loop has run for real.
How to read this backlog
Priority here is sequencing, not importance. The order follows the principles: prove the manual version first, automate only where friction is real, and add the largest builds last. When v2 of the Project OS is scoped, this list is the starting point, and any item can be repriced against the evidence by then.
